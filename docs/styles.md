# Styles Reference

A practical reference for the SCSS architecture used in `opencode-model-gui`. For the original migration design and full token rationale, see [`scss-refactor-plan.md`](./scss-refactor-plan.md).

---

## 1. Overview

- **Preprocessor:** Dart Sass (`sass` devDependency), compiled by Vite.
- **Architecture:** SCSS 7-1, rooted at `src/styles/`.
- **Entry point:** `src/styles/main.scss` — the only file imported from JS.
- **JS import:** `src/main.jsx` contains a single line: `import './styles/main.scss'`.
- **CSS custom properties** declared on `:root` in `src/styles/base/_root.scss` are still used at runtime so inline JSX styles can reference them via `var(--…)`.

---

## 2. Folder Structure

```
src/styles/
├── main.scss              # Entry — only @use statements
├── abstracts/             # Tokens & helpers — no CSS output
├── base/                  # Reset, :root custom properties, typography, base elements
├── themes/                # Theme overrides layered on top of :root
├── layout/                # App shell, sidebars, grids, headers, footers
├── components/            # One partial per UI component
├── pages/                 # Page-scoped rules
├── utilities/             # Single-purpose generated helper classes
└── vendors/               # Third-party imports (currently empty)
```

Every folder has an `_index.scss` barrel that `@forward`s its siblings. Partials are prefixed with `_`.

---

## 3. `main.scss` and `@use` Order

```scss
@use "vendors";
@use "abstracts";
@use "base";
@use "themes";
@use "layout";
@use "components";
@use "pages";
@use "utilities";
```

Order matters:

- **vendors → abstracts → base** loads tokens and global element rules before anything that depends on them.
- **themes** overrides `:root` after the base declares it.
- **layout → components → pages** moves from broad scaffolding to specific UI.
- **utilities** comes last so single-purpose classes win the cascade against components.

## 4. Tokens

All design tokens live in `src/styles/abstracts/_variables.scss` and `_maps.scss`. Treat those files as the source of truth — don't duplicate values elsewhere.

| Category | Source | Access via |
|---|---|---|
| Colors (21) | `$colors` map | `color('name')` |
| Spacing (17 steps) | `$spacing` map | `spacing('key')` |
| Typography presets | `$typography-presets` | `@include typography('preset')` |
| Breakpoints (sm, md, lg, xl, 2xl) | `$breakpoints` map | `@include breakpoint('name')` |
| Radii | `$radii` map | `radius('key')` |
| Shadows | `$shadows` map | `shadow('key')` |
| Z-index layers | `$z-layers` map | `z('layer')` |

Component CSS that needs runtime values (theme-driven) continues to read the `var(--…)` custom properties declared in `base/_root.scss`.

---

## 5. Functions

Every function `@error`s on an unknown key so typos fail loudly at build time.

```scss
spacing('4')       // → 1rem        (lookup in $spacing)
color('accent-blue')   // → #a3c9ff
rem(24)            // → 1.5rem      (px → rem, 16px root)
z('overlay')       // → number      (lookup in $z-layers)
shadow('card')     // → var(--shadow-card)
radius('xl')       // → token from $radii
```

`breakpoint-value($key)` is an internal helper used by the `breakpoint()` mixin; call the mixin, not the function.

## 6. Mixins

```scss
// Responsive — min-width media query
@include breakpoint('md') { /* … */ }

// Layout
@include flex(row, center, center, spacing('3'));
@include grid(3, spacing('4'));

// Accessibility & patterns
@include focus-ring;                    // 2px outer ring, accent blue
@include truncate;                      // single-line ellipsis
@include visually-hidden;               // screen-reader-only

// Motion
@include transition(background, color); // defaults to background/color/border-color

// Typography
@include typography('title');           // applies preset from $typography-presets

// Resets & surfaces
@include button-reset;
@include card-surface;                  // shared card background/border/radius/shadow
@include scrollbar-dark;                // webkit dark scrollbar
```

## 7. Utility Classes

Utilities are generated by `@each` loops over the abstracts maps, so adding a token automatically extends the utility set. Naming follows a Tailwind-style convention.

| Grid | `.grid`, `.grid-cols-3`, `.col-span-2`, `.gap-4` | 37 |

| Sizing | `.w-full`, `.h-screen`, `.w-auto` | 11 |
| Typography | `.text-lg`, `.font-semibold`, `.text-center` | 34 |

## 8. Writing a New Component Partial

```scss
// src/styles/components/_my-widget.scss
@use "../abstracts" as *;

.my-widget {
  @include card-surface;
  padding: spacing('4');
  @include flex(column, flex-start, stretch, spacing('2'));

  &__title {
    @include typography('title');
  }

  &--compact {
    padding: spacing('2');
  }

  &:hover {
    @include transition(background);
    background: color('bg-elevated');
  }
}
```

Steps:

1. Create `src/styles/components/_my-widget.scss`.
2. First line: `@use "../abstracts" as *;` so `color()`, `spacing()`, mixins resolve.
3. Add `@forward "my-widget";` to `src/styles/components/_index.scss`.
4. Use utility classes in JSX for one-off layout (`className="flex gap-3"`); use a component partial when the rule recurs or has state/modifiers.

### Nesting rules

- **Max depth: 3 levels.**
- Use `&` for pseudo-classes (`&:hover`, `&:focus`), state classes (`&.is-active`), and BEM modifiers/elements (`&--primary`, `&__icon`).
- Do **not** nest descendant selectors more than two levels deep.
- Do **not** nest media queries inside selectors more than once.

## 9. CSS Custom Properties (`:root`)

`src/styles/base/_root.scss` declares the `--…` custom properties that JSX inline styles read at runtime. The following are referenced from `.jsx` files today:

- `--text-dim`
- `--text-primary`
- `--text-secondary`
- `--accent-green`
- `--font-mono`
- `--bg-card`

**Rule:** if you add a `--…` token that JSX will read via `var(--…)`, declare it in `_root.scss`. The SCSS scalars in `_variables.scss` are not visible to runtime JS.

## 10. Build & Dev

```bash
npm install        # sass is already a devDependency
npm run dev        # Vite dev server with HMR for SCSS
npm run build      # production build
```

Vite handles SCSS automatically because `sass` is installed. No `vite.config.js` changes are required.

You may see a `DEPRECATION WARNING [legacy-js-api]` from Sass during builds. This is emitted by Vite's internal Sass integration, not by project code; it resolves when Vite upgrades its Sass bridge.

## 11. Known Limitations & Follow-ups

- **Hardcoded hex values** remain in a few component partials. They are candidates for migration to `color()` lookups once added to the palette:
  - `#c0392b` in `components/_ollama-status-card.scss`, `components/_titlebar.scss`, `layout/_status-bar.scss` (already mirrored as `$red-danger` in `_variables.scss` but not yet exposed via the `$colors` map).
  - `#ffffff` in `components/_toggle.scss`.
- **`color-mix(in srgb, var(--chip-color, #59a6ff) …)`** fallbacks in `components/_agent-card.scss` must remain literal — the CSS `var()` fallback grammar does not accept SCSS function calls.
- **`legacy-js-api` warning** — see Build & Dev above; not actionable in project code.
